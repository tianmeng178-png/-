# AI 驱动热交换器智能设计系统 (V1) - 完整运行逻辑报告

## 一、系统架构概述

### 1.1 技术栈
- **前端框架**：React + TypeScript + Vite + Ant Design
- **后端框架**：FastAPI (Python)
- **通信协议**：WebSocket (实时) + REST API
- **计算引擎**：OpenFOAM v11 (通过 WSL Ubuntu-24.04)
- **数据持久化**：JSON 文件存储

### 1.2 核心模块
```
heat_exchanger_ai/
├── backend/          # 后端服务
│   ├── models/       # 数据模型定义
│   ├── services/     # 业务逻辑服务
│   ├── websocket/    # WebSocket 连接管理
│   └── utils/        # 工具函数
├── frontend/         # 前端应用
│   ├── src/
│   │   ├── components/  # React 组件
│   │   ├── services/    # API 服务
│   │   └── stores/      # 状态管理
└── src/              # OpenFOAM 控制器 (独立模块)
```

## 二、系统启动流程

### 2.1 后端启动 (main.py)

**步骤详细说明：**

1. **导入依赖与服务初始化**
   - 导入 FastAPI、CORS 中间件、WebSocket 支持
   - 初始化核心服务实例：
     - `SimulationManager` - 仿真流程管理
     - `OpenFOAMService` - OpenFOAM 集成
     - `LLMService` - 自然语言处理（预留）
     - `SimulationHistory` - 历史记录管理
     - `ParameterPresets` - 参数预设管理
     - `ConnectionManager` - WebSocket 连接管理

2. **生命周期管理**
   ```python
   @asynccontextmanager
   async def lifespan(app: FastAPI):
       # 启动时：设置进度回调、创建数据目录
       simulation_manager.set_progress_callback(progress_callback)
       os.makedirs("static", exist_ok=True)
       os.makedirs("data/simulations", exist_ok=True)
       os.makedirs("data/presets", exist_ok=True)
       yield
       # 关闭时：清理所有 WebSocket 连接
   ```

3. **OpenFOAM 可用性检查**
   - `OpenFOAMService._check_availability()` 自动执行
   - 检查路径：`\\wsl.localhost\Ubuntu-24.04\opt\openfoam11`
   - 或通过 `wsl -d Ubuntu-24.04 which foamRun` 验证

4. **CORS 配置**
   - 允许来源：`localhost:3000`, `localhost:5173`, `*`
   - 允许所有方法和头

5. **启动 Uvicorn 服务器**
   - 监听地址：`0.0.0.0:8000`
   - 启用热重载：`reload=True`

### 2.2 前端启动 (Vite)

**步骤详细说明：**

1. **加载环境变量**
   - 从 `.env` 读取配置
   - `VITE_API_URL=http://localhost:8000/api`
   - `VITE_WS_PORT=8000`

2. **初始化 React 应用**
   - 挂载 `App.tsx` 到 `index.html#root`
   - 初始化 Zustand 状态管理 (`useStore`)
   - 配置 Ant Design 主题

3. **WebSocket 连接建立**
   - 等待用户点击"开始仿真"后连接
   - 连接 URL：`ws://localhost:8000/ws/simulation/{simulation_id}`

## 三、用户交互流程详解

### 3.1 参数配置阶段

**前端组件：** `ParameterPanel.tsx`

**详细流程：**

1. **用户选择预设参数**
   - 从 `presets.json` 加载默认配置
   - 可选项：
     - 水冷铜基板标准配置
     - 高功率散热配置
     - 风冷配置

2. **自定义参数设置**
   支持的参数类型：

   | 参数类别 | 字段 | 单位 | 范围说明 |
   |---------|------|------|---------|
   | **几何参数** | `channel_width` | m | 1μm - 1cm |
   | | `channel_height` | m | 1μm - 1cm |
   | | `channel_length` | m | - |
   | | `channel_count` | - | 1 - 1000 |
   | | `wall_thickness` | m | 1μm - 1cm |
   | **流动参数** | `inlet_velocity` | m/s | ≤ 100 m/s |
   | | `inlet_temperature` | K | - |
   | | `outlet_pressure` | Pa | - |
   | **热参数** | `heat_flux` | W/m² | ≤ 10 MW/m² |
   | | `base_temperature` | K | - |
   | **材料参数** | `fluid_type` | - | water/air/ethylene_glycol/engine_oil |
   | | `solid_material` | - | copper/aluminum/steel/silicon |
   | **求解参数** | `mesh_resolution` | - | - |
   | | `convergence_criteria` | - | - |
   | | `max_iterations` | - | - |
   | **仿真模式** | `simulation_mode` | - | mock/openfoam |

3. **参数验证（前端 + 后端）**
   - 前端实时验证：基本类型检查
   - 后端完整验证：调用 `/api/validate-parameters`
   - 验证内容：
     - 尺寸范围（微尺度约束）
     - 速度限制（≤ 100 m/s）
     - 热通量限制（≤ 10 MW/m²）
     - 纵横比警告（建议 2-5）
     - 雷诺数计算与警告

### 3.2 仿真启动流程

**详细时序图：**

```
用户点击"开始仿真"
    ↓
前端: 收集参数 → POST /api/simulation/start
    ↓
后端 main.py: start_simulation() 端点
    ├─ 1. 打印接收到的参数（调试）
    ├─ 2. 调用 simulation_manager.validate_parameters()
    │   └─ 基础验证（当前使用，禁用了 parameter_constraints）
    │       ├─ 检查尺寸范围
    │       ├─ 检查速度限制
    │       ├─ 检查热通量限制
    │       └─ 计算纵横比和雷诺数警告
    ├─ 3. 生成 UUID simulation_id
    ├─ 4. 保存到 simulation_history (JSON)
    └─ 5. 调用 simulation_manager.start_simulation()
        ↓
    SimulationManager.start_simulation()
        ├─ 1. 提取 simulation_mode 参数
        ├─ 2. 初始化仿真状态对象
        │   ├─ status: RUNNING
        │   ├─ progress: 0
        │   ├─ current_step: "初始化仿真环境"
        │   ├─ log_messages: []
        │   └─ reynolds_number: 预计算
        ├─ 3. 根据模式选择任务
        │   ├─ mock: _simulate_progress() (异步任务)
        │   └─ openfoam: _run_openfoam_simulation() (异步任务)
        └─ 4. 保存任务到 self.simulation_tasks
    ↓
返回给前端: simulation_id + websocket_url
    ↓
前端: 建立 WebSocket 连接
    ↓
WebSocket 握手:
    ├─ 前端发送: {"type": "subscribe"}
    └─ 后端返回: 初始状态更新
```

## 四、仿真执行流程（两种模式）

### 4.1 模拟模式 (Mock Mode)

**执行函数：** `SimulationManager._simulate_progress()`

**详细步骤：**

1. **预定义步骤序列**
   ```python
   steps = [
       ("初始化仿真环境", 10),
       ("生成计算网格", 25),
       ("设置边界条件", 40),
       ("运行CFD求解器", 60),
       ("计算流动场", 75),
       ("计算温度场", 90),
       ("生成结果报告", 100)
   ]
   ```

2. **进度更新循环**
   - 每个步骤等待 2 秒
   - 更新 `current_step` 和 `progress`
   - 追加日志消息
   - 通过 `progress_callback` 发送 WebSocket 消息

3. **完成时计算性能指标**
   - 调用 `_calculate_performance_metrics()`
   - 基于工程公式计算：
     - 雷诺数 (Re)
     - 努塞尔数 (Nu)
     - 传热系数 (h)
     - 压力降 (ΔP)
     - 最高温度 (T_max)
     - 热阻 (R)
     - 效率 (η)

4. **发送完成消息**
   - WebSocket: `{"type": "completed", "data": {...}}`
   - 包含性能指标

### 4.2 OpenFOAM 模式 (Real Mode)

**执行函数：** `SimulationManager._run_openfoam_simulation()`

**详细步骤：**

1. **阶段 1：初始化 OpenFOAM 环境 (10%)**
   - 更新状态
   - 发送进度更新
   - 等待 0.5 秒

2. **阶段 2：创建仿真案例 (25%)**
   - 创建临时目录：`tempfile.mkdtemp(prefix="openfoam_case_")`
   - 尝试从模板创建：`openfoam_templates/microchannel/`
   - 前端参数转换：
     ```python
     controller_params = {
         "velocity": params.inlet_velocity,
         "inlet_temperature": params.inlet_temperature - 273.15,  # K → °C
         "wall_temperature": 80.0,
         "channel_width": params.channel_width * 1e6,        # m → μm
         "channel_height": params.channel_height * 1e6       # m → μm
     }
     ```
   - 创建 `OpenFOAMController`（来自 `src/foam_controller.py`）

3. **阶段 3：生成计算网格 (40%)**
   - 尝试调用 `controller.run_mesh()`
   - 在独立线程中执行：`asyncio.to_thread()`
   - 记录成功/失败日志

4. **阶段 4：运行 CFD 求解器 (60%)**
   - 尝试调用 `controller.run_simulation()`
   - 等待 1 秒（模拟执行）
   - 记录成功/失败日志

5. **阶段 5：计算流动场、温度场、生成报告 (75%, 90%, 100%)**
   - 每个阶段等待 1.5 秒
   - 更新状态和日志
   - 发送 WebSocket 进度

6. **阶段 6：提取结果**
   - 尝试调用 `controller.get_results()`
   - 如果失败，使用模拟结果公式
   - 构建 `PerformanceMetrics` 对象

7. **阶段 7：仿真完成**
   - 标记状态为 `COMPLETED`
   - 记录结束时间
   - 发送完成 WebSocket 消息
   - 清理临时目录：`shutil.rmtree(temp_dir)`

## 五、WebSocket 实时通信

### 5.1 WebSocket 端点

**URL 格式：** `ws://localhost:8000/ws/simulation/{simulation_id}`

**后端处理：** `main.py.websocket_endpoint()`

### 5.2 消息类型

| 消息类型 | 方向 | 数据结构 | 用途 |
|---------|------|---------|------|
| **subscribe** | 前端→后端 | `{"type": "subscribe"}` | 订阅仿真更新 |
| **pong** | 前端→后端 | `{"type": "pong"}` | 心跳响应 |
| **control** | 前端→后端 | `{"type": "control", "command": "pause/resume/stop"}` | 控制命令 |
| **get_status** | 前端→后端 | `{"type": "get_status"}` | 请求当前状态 |
| **status_update** | 后端→前端 | `{"type": "status_update", "data": {...}}` | 状态更新 |
| **progress** | 后端→前端 | `{"type": "progress", "data": {...}}` | 进度更新 |
| **completed** | 后端→前端 | `{"type": "completed", "data": {...}}` | 完成通知 |
| **error** | 后端→前端 | `{"type": "error", "data": {...}}` | 错误通知 |

### 5.3 连接管理

**管理器：** `ConnectionManager` (`websocket/connection_manager.py`)

**功能：**
- 跟踪活跃连接数
- 管理每个仿真的订阅者
- 广播消息给所有订阅者
- 处理连接断开和清理

### 7.2 潜在问题与风险

#### **问题 7.2.1：OpenFOAM 真实调用可能失败**

**症状：**
- OpenFOAM 模式选择后，仿真可能在某个阶段失败
- 日志显示："模板案例不可用，使用简化 OpenFOAM 仿真"

**根本原因：**
- `openfoam_templates/microchannel/` 目录可能不存在或不完整
- `src/foam_controller.py` 可能有未处理的异常
- WSL 路径访问权限问题

**解决方案：**
1. 确保 `openfoam_templates/microchannel/` 包含完整的 OpenFOAM 案例结构
2. 实现更健壮的错误处理和备用方案
3. 检查 WSL 文件系统访问权限
4. 添加 OpenFOAM 案例创建前的预检查

---

#### **问题 7.2.2：WebSocket 连接不稳定**

**症状：**
- 前端显示"WebSocket 未连接"
- 进度更新丢失
- 浏览器控制台显示 WebSocket 断开

**根本原因：**
- 网络超时
- 后端服务重启
- 浏览器标签页休眠
- 防火墙阻止连接

**解决方案：**
1. 实现 WebSocket 自动重连机制
2. 添加心跳检测（ping/pong）
3. 实现状态恢复：断开重连后从 `/api/simulation/{id}/status` 获取最新状态
4. 增加连接超时和重试逻辑

---

#### **问题 7.2.3：参数验证过于严格**

**症状：**
- 用户输入的合理参数被拒绝
- 警告信息过多影响用户体验

**根本原因：**
- 硬编码的参数范围可能不符合实际工程需求
- 警告阈值设置过于保守

**解决方案：**
1. 将验证参数配置化（从 JSON/YAML 加载）
2. 区分"错误"（必须修复）和"警告"（建议）
3. 允许用户覆盖警告继续仿真
4. 添加参数验证级别配置

---

#### **问题 7.2.4：OpenFOAM 执行超时**

**症状：**
- 仿真卡在"运行 CFD 求解器"阶段
- 长时间无响应
- 最终超时失败

**根本原因：**
- 网格过于精细导致计算时间过长
- 求解器发散或收敛困难
- WSL 性能瓶颈
- 缺少超时和中断机制

**解决方案：**
1. 实现仿真任务超时机制
2. 添加求解器收敛监控
3. 提供网格质量警告
4. 实现仿真取消功能
5. 添加性能预估和时间警告

---

#### **问题 7.2.5：前后端数据类型不一致**

**症状：**
- 数值精度丢失
- 枚举值不匹配
- 日期时间解析错误

**根本原因：**
- TypeScript → Python 类型转换问题
- 单位转换不一致（K/°C, m/μm）
- JSON 序列化/反序列化

**解决方案：**
1. 实现严格的类型验证层
2. 统一单位规范（文档化每个字段的单位）
3. 添加类型转换中间件
4. 实现端到端类型测试

---

#### **问题 7.2.6：数据持久化可靠性**

**症状：**
- 仿真历史记录丢失
- JSON 文件损坏
- 并发写入冲突

**根本原因：**
- 缺少事务机制
- 无数据备份
- 多进程/多线程写入冲突

**解决方案：**
1. 使用数据库替代 JSON 文件（SQLite/PostgreSQL）
2. 实现文件写入原子性
3. 添加数据备份和恢复功能
4. 实现写锁机制

---

#### **问题 7.2.7：GPU 加速未实现**

**症状：**
- `use_gpu_acceleration` 参数存在但无实际效果
- OpenFOAM 未使用 NVIDIA 4060 GPU

**根本原因：**
- OpenFOAM 默认编译版本不支持 GPU
- 缺少 CUDA/OpenACC 配置
- 无 GPU 可用性检测

**解决方案：**
1. 编译支持 GPU 的 OpenFOAM 版本（使用 OpenFOAM-GPU 或通过 Foam-Extend）
2. 实现 GPU 可用性检测
3. 添加 GPU 加速配置选项
4. 实现 GPU 性能监控

---

#### **问题 7.2.8：CORS 配置过于宽松**

**症状：**
- 安全风险：允许所有来源访问

2. 生产环境严格限制允许的来源
3. 添加 Origin 白名单
4. 实现请求签名验证

---

#### **问题 7.2.9：缺少日志记录和监控**

**症状：**
- 问题排查困难
- 无性能指标
- 无法追踪用户行为

**根本原因：**
- 只有简单的 print 语句

- 缺少性能监控

1. 集成日志库（logging/structlog）

4. 实现错误追踪和告警

#### **问题 7.2.10：前端状态管理复杂度**

**症状：**
- Zustand 状态同步问题
- 组件重新渲染过多
- 状态更新延迟

**根本原因：**

**解决方案：**

2. 使用状态选择器避免不必要重渲染
